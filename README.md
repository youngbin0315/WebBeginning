# 순수 HTML/CSS/JS 기반 반응형 포트폴리오 웹사이트

외부 프레임워크나 라이브러리(React, Tailwind, jQuery 등) 없이, 순수 웹 표준 기술(**Vanilla 웹스택**)만을 사용하여 구현한 인터랙티브 반응형 포트폴리오입니다. 웹의 핵심 동작 원리인 **"사용자 이벤트 → 상태 변경 → DOM 조작"**의 흐름을 직접 제어하도록 설계되었습니다.

---

## 사용 기술 및 개발 환경
- **Language & Markup:** HTML5 (Semantic Markup), CSS3, JavaScript (ES6+)
- **Environment & Tools:** VS Code, Live Server, Git / GitHub Pages
- **API 연동:** GitHub REST API

---

## 핵심 기능 및 구현 세부 사양
1. **반응형 레이아웃 (모바일 퍼스트)**
   - `Flexbox`를 활용하여 글로벌 네비게이션바 레이아웃 구조화.
   - `Grid`의 `auto-fit` 및 `minmax(300px, 1fr)`를 사용하여 Projects 카드가 디바이스 너비에 맞춰 유연하게 변형되도록 구현.
   - 768px(태블릿), 1024px(데스크톱) 미디어나 조건절 설계를 통해 모바일 햄버거 메뉴 토글 완성.
2. **다크 모드 & 상태 유지**
   - CSS 변수(`:root`, `[data-theme="dark"]`) 분리 제어.
   - 테마 상태를 `localStorage`에 저장하여 새로고침 후에도 사용자의 이전 설정 유지.
3. **인터랙티브 UI 및 UX**
   - **네비게이션 배경 전환:** 스크롤 위치가 `60px` 이상일 때 투명 배경에서 블러/반투명 배경으로 전환.
   - **스크롤 탑 버튼:** 스크롤 위치 `300px`를 기준으로 점진적 표시(`opacity`) 처리 및 클릭 시 부드러운 상단 스크롤 작동.
   - **스크롤 애니메이션:** `Intersection Observer API`를 활용하여 임계값 `0.2` 조건 충족 시 섹션 순차적 등장 효과 부여.
4. **GitHub API 동적 연동 및 예외 상태 처리**
   - `fetch` 및 `async/await` 구조로 비동기 통신 구현.
   - 요청 진행 상태에 따른 **4가지 UI 상태 분기** 관리 (로딩 스피너 / 성공 카드 리스트 렌더링 / 에러 발생 시 재시도 버튼 제공 / 데이터 공백 상태 안내).
5. **폼 유효성 검사 (Form Validation)**
   - 이메일 정규표현식 검증 로직 및 공백 검사.
   - `preventDefault()` 제어로 불필요한 브라우저 리로드를 막고 에러 메시지를 실시간 인라인으로 출력.

### ❓ Q1. HTML 시맨틱 태그는 왜 사용하고, 이 프로젝트는 어떤 기준으로 구조를 설계했나요?
> **답변 핵심**
> 시맨틱 태그는 브라우저와 웹 검색 엔진, 그리고 스크린 리더(시각장애인용)에게 **웹페이지의 구조적 의미를 명확하게 전달**하기 위해 사용합니다. 이는 웹 표준과 웹 접근성(WCAG)을 준수하기 위한 가장 기본적이고 강력한 첫걸음입니다.

* **설계 기준:** 무분별한 `<div>` 폭탄을 배제하고 다음과 같은 논리적 흐름으로 설계했습니다.
    * **`<header>` & `<nav>`:** 상단 글로벌 레이아웃 및 내비게이션 메뉴 영역 분리
    * **`<main>`:** 본문의 가장 핵심적인 콘텐츠 영역 선언
    * **`<section>`:** 논리적으로 독립된 콘텐츠 단락 구분 (`#hero`, `#about`, `#skills`, `#projects`, `#contact`)
    * **`<article>`:** Projects 섹션 내에서 동적으로 생성되는 개별 카드형 데이터 아이템 매핑
    * **`<footer>`:** 최하단 저작권 정보 및 소셜 링크 마감

### ❓ Q2. CSS에서 Flexbox와 Grid의 결정적인 차이점은 무엇인가요? 언제 각각을 선택해야 하죠?
> **답변 핵심**
> 가장 큰 차이는 **차원의 수**입니다. 
> * **Flexbox**는 **1차원(단일 방향) 레이아웃 시스템**으로 가로 혹은 세로 중 한 방향 정렬에 강력합니다.
> * **Grid**는 **2차원(행과 열) 레이아웃 시스템**으로 가로와 세로를 동시에 정교한 바둑판 격자 형태로 배치할 때 최고의 효율을 보입니다.

* **본 프로젝트에서의 선택 기준:**
    * **Flexbox 선택:** 상단 헤더(`nav-container`)처럼 좌측 로고와 우측 메뉴를 단일 선상(가로축)에 나란히 배치하고 균등 정렬할 때 사용했습니다.
    * **Grid 선택:** Projects 구역처럼 동적인 카드가 바둑판 모양으로 정렬되면서, 화면 너비에 따라 자동으로 줄바꿈이 일어나는 반응형 구조를 미디어 쿼리 없이 구현하기 위해 `grid-template-columns: repeat(auto-fit, minmax(320px, 1fr));` 문법을 선택했습니다.

### ❓ Q3. `querySelector`와 `addEventListener`를 사용한 DOM 이벤트 연결 흐름을 설명해 주세요.
> **답변 핵심**
> HTML 태그 내부에 인라인으로 이벤트를 작성하는 방식(`onclick="..."`)은 마크업과 스크립트가 뒤엉켜 유지보수와 관심사 분리 관점에서 지양해야 합니다. 

* **이벤트 연결 흐름:**
    1.  자바스크립트 영역에서 `document.querySelector` 계열 메서드를 사용하여 제어하고자 하는 HTML 요소를 정확하게 추적 및 탐색하여 메모리에 캐싱합니다.
    2.  수집된 요소 노드에 `.addEventListener('이벤트종류', 콜백함수)`를 바인딩합니다.
    3.  이로써 HTML 구조(Structure)와 동적 스크립트 로직(Behavior)이 완벽히 분리되어 안전하고 단단한 이벤트 구동 환경이 완성됩니다.

### ❓ Q4. ES6+ 문법(화살표 함수, 구조분해 할당, map)이 왜 필요하고 이 프로젝트에서 어떻게 사용했나요?
> **답변 핵심**
> 최신 자바스크립트 문법은 **코드를 획기적으로 압축하고 가독성을 높여** 인간의 실수로 인한 에러 유발 가능성을 줄이기 위해 필수적입니다.

* **프로젝트 내 활용 사례:**
    * **화살표 함수 (`=>`):** 불필요한 `function` 키워드를 생략하여 코드를 간결하게 만들고, 의도치 않은 `this` 바인딩 문제를 예방했습니다.
    * **구조분해 할당 (`destructuring`):** GitHub API 서버에서 받아온 무거운 레포지토리 객체에서 필요한 알맹이 변수(`const { name, description, html_url } = project;`)만 깔끔하게 추출하여 사용했습니다.
    * **`map` 메서드:** 받아온 API 데이터 배열을 순회하며 원시 데이터를 똑같은 규격의 HTML 문자열 템플릿 카드로 일괄 변환·치환하는 프론트엔드 핵심 조작 기법을 구사했습니다.

### ❓ Q5. `fetch`와 `async/await` 비동기 처리 과정 및 4가지 UI 상태 표현법은 무엇인가요?
> **답변 핵심**
> 외부 서버에서 데이터를 가져올 때 브라우저의 메인 스레드가 멈춰 서는 현상(블로킹)을 방지하기 위한 비동기 통신 표준 기술입니다. 복잡한 콜백 헬(Callback Hell)이나 `.then()` 체이닝의 가독성 저해를 해결하기 위해, 직관적인 동기식 코드처럼 읽히는 `async/await` 문법을 사용했으며 `try/catch` 블록으로 예외 처리를 마감했습니다.

* **4가지 UI 상태 표현 기법 (UX 최적화):**
    1.  **로딩 상태 (`loading`):** 서버 응답 대기 중 "🔄 로딩 중..." 텍스트를 화면에 띄워 사용자 이탈 방지
    2.  **성공 상태 (`success`):** 데이터를 정상 수신하면 `map()` 메서드로 가공한 카드 리스트 렌더링
    3.  **에러 상태 (`error`):** 네트워크 장애나 API 제한 시 경고 메시지와 함께 **[다시 시도하기]** 버튼 제공
    4.  **빈 상태 (`empty`):** 받아온 데이터 배열의 길이가 `0`일 때 "📂 표시할 프로젝트가 없습니다"라는 안내 문구 출력

### ❓ Q6. "이벤트 ➔ 상태 변경 ➔ 화면 업데이트"라는 단방향 데이터 흐름이 어떻게 연결되나요? (React의 기초)
> **답변 핵심**
> Vanilla JS로 웹을 만들 때 DOM을 수동으로 일일이 직접 조작하는 방식은 규모가 커질수록 코드가 꼬이기 쉽습니다. 모던 프론트엔드는 **"데이터 상태(State)가 변하면 화면(UI)은 자동으로 이에 맞춰 변한다"**라는 철학을 따릅니다.

* **단방향 연결 메커니즘 (예시: 다크 모드 토글):**
    1.  **이벤트 발생:** 사용자가 테마 전환 버튼을 클릭합니다.
    2.  **상태 변경 (State Update):** 자바스크립트 내 중앙 제어 객체인 `state.theme` 데이터 값을 변경합니다 (`light` ➔ `dark`).
    3.  **화면 업데이트 (Render):** 화면을 직접 뒤엎는 대신, 바뀐 최신 데이터 상태를 바라보고 화면을 다시 그려주는 독립 함수인 `applyTheme()`을 호출하여 HTML의 `data-theme` 속성을 싱크업합니다.
    * 이 개념은 향후 모던 프레임워크인 **React의 상태 기반 렌더링(State-driven rendering)**과 단방향 데이터 흐름을 이해하는 핵심 근간이 됩니다.
